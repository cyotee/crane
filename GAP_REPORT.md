## LR-1: NatSpec Documentation Standard (Mandatory & Verifiable)

**Canonical Reference:** ERC8023 suite (`IMultiStepOwnable.sol`, `MultiStepOwnableTarget.sol`, `MultiStepOwnableRepo.sol`, `MultiStepOwnableFacet.sol`) – full `// tag::...[]` / `// end::...[]` + `@custom:interfaceid`, `@custom:selector`, `@custom:topiczero`, rich `@notice`/`@param`/`@custom:emits`/`@custom:throws`.

### Gaps
- **Coverage is highly uneven.** Full or near-full tagging + custom tags exist only in small pockets (ERC8023 area, a few interfaces like `IPermit2Aware`, `BetterIERC20`, some exchange interfaces, and recent factory files such as `IDiamondFactoryPackage.sol`, `ICreate3Factory.sol`, `Create3FactoryDFPkg.sol`).
- Vast majority of the codebase lacks the required style:
  - Most factories (`Create3Factory*.sol`, `DiamondPackageCallBackFactory*.sol`, registries).
  - Introspection (ERC165/ERC2535/ERC8109 Repos, Facets, Targets).
  - Tokens (ERC20/ERC721/ERC1155 Repos, Targets, Facets, DFPkgs).
  - All protocol ports (AwareRepos, Services, Facets).
  - Utilities (Sets, math, crypto, etc.).
  - Many interfaces and all DFPkgs beyond the few mentioned.
- **No verification script exists.** No Foundry Script (or any committed tool) that outputs compiler-computed `interfaceId` / `selector` / event topic0 values for pasting into tags. Current practice relies on ad-hoc `cast` or manual values in a few places.
- **Tests are almost entirely undocumented.** Scope explicitly includes tests (`*.t.sol`, TestBases, Behaviors, handlers). Only ERC8023 TestBase/Handler files have partial tags; everything else has none.
- Many existing `@custom:selector` / `@custom:interfaceid` values are present but not wrapped with the required tag style and not systematically verified.

**Impact:** Blocks professional documentation extraction and agent consumption.

## LR-7: Testing Standards

**Positive observations:** Some good usage exists (Behavior_IFacet tests, some ERC721 Behavior usage, TestBase patterns in a few places, handler-based invariants in ERC20 area).

### Gaps (many violations of the non-exhaustive list)
- **Initialization failures:** Several tests (especially fork and protocol ports) pass `address(0)` for implementations or leave DFPkg facets uninitialized. DFPkg tests frequently do not assert all facets are non-zero.
- **Weak assertions:** Common pattern of “balance changed” or “something happened” instead of exact delta. Preview functions are rarely (if ever) asserted for exact parity against the executed action.
- **Missing declaration tests:**
  - Very few tests assert that a deployed Facet returns the expected `facetInterfaces()` / `facetFuncs()`.
  - Package tests rarely (or never) fully assert `facetAddresses()`, `facetCuts()`, `diamondConfig()`, `packageName()`, `calcSalt`, etc.
- **Behavior library usage is inconsistent and incomplete.** `Behavior_IFacet` is used in dedicated behavior tests but not applied universally to actual Facets. `Behavior_IDiamondFactoryPackage` usage is minimal or absent. Many protocol and standard tests bypass the Behaviors entirely.
- Missing or weak coverage of:
  - Full DFPkg lifecycle (especially `initAccount` via real factory delegatecall).
  - Registry population after deployment.
  - CREATE3/salt determinism with explicit assertions.
  - Storage isolation across multiple instances.
  - Reentrancy + full access-control matrices using the actual Repos.
  - Fork parity for major ports.
- TestBase/CraneTest inheritance is sometimes bypassed or incomplete.
- NatSpec on test code is almost nonexistent (violates LR-1 scope).
