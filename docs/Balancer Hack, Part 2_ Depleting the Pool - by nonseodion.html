In Part 1, we came up with this attack plan:

  1. Use batchSwap() to borrow BPT. Swap BPT tokens for the underlying pool tokens, driving the pool to low liquidity. We now have tokens and owe the Vault BPT.

  2. Exploit the rounding at low liquidity. Perform a _swapGivenOut() swap that triggers the rounding error in _upscale(), corrupting the invariant downward.

  3. Repay the BPT debt — at the new, lower rate. Because the invariant has fallen, we need fewer underlying tokens to pay back the BPT we owe.

  4. Pocket the difference. We exit the batch having spent fewer tokens than we received.

We can rename the steps, in order, as: deplete liquidity, exploit the rounding bug, refill liquidity, and withdraw. This piece will cover Step 1: deplete liquidity.

What makes Step 1 particularly important is that it creates the environment our bug needs to grow. And we are responsible parents, are we not? So let’s take care of our baby.

A Few Things We Need to Get Right First

MEV Consideration

Even as a hacker, you can still have your exploit hijacked before it is executed. We have enemies lurking in the shadows called Maximal Extractable Value (MEV).

MEV bots scan transactions to see if they can reorder them and extract profit. For example, if you create a transaction to withdraw from a contract with no access restriction, your transaction may fail—and when you check the explorer, you’ll see a bot has already withdrawn the funds.

In our case, if a bot detects that our transaction is profitable and requires no special privileges, it can copy the transaction data and execute its own version before ours. This may cause our transaction to fail.

To prevent this, the hacker separated the withdrawal from the exploit. We’ll do the same.

Separate Withdrawal Transaction

Our attack transaction will not include the withdrawal.

If a bot sees that we withdraw a large amount at the end of our transaction, it may attempt to hijack the exploit. Instead, we’ll send the funds to our internal balance on Balancer and withdraw them in a separate transaction.

A bot cannot hijack the withdrawal because it requires privileged access to our internal balance—access only we have.

That should protect us from MEV.

Pool Tokens

Another important detail is the arrangement of tokens in the pool. ComposableStablePools allow users to swap the liquidity provider token (BPT) for any token in the pool.

We’ll be targeting the wETH/osETH pool. Its token array is arranged as follows:

[wETH, BPT, osETH]

This means BPT has an index of 1. You’ll see this used multiple times in the code.

Fees

Balancer collects two types of fees: protocol fees and liquidity provider fees. To understand the fees, we need to know the types of transactions Balancer performs. There are three types of transactions:

  • Swaps: These are the regular swaps done by AMMs. These transactions exchange an underlying token for another underlying token. Underlying tokens are tokens deposited by a liquidity provider, e.g., wETH and osETH.

  • Proportional Joins/Exits: A join is done when a liquidity provider adds underlying tokens to the pool and receives liquidity provider tokens in the form of Balancer Pool Tokens (BPT). An exit is done when a liquidity provider removes underlying tokens from the pool and burns their BPT. If they join or exit with a proportional amount of underlying tokens, it is a regular join/exit. E.g., if a user swaps a proportional amount of wETH and osETH for BPT in a wETH/osETH pool, it is a proportional join transaction.

  • Non-proportional Joins/Exits: When a liquidity provider does not supply a proportional amount of tokens in a join or exit transaction, it is a non-proportional join or exit. This transaction can be described as a swap with BPT. E.g., if a user swaps only wETH for BPT in a wETH/osETH pool, it is a non-proportional join transaction.

The original hacker may not have done this, but this will ensure that our code is correct by predicting the amount the protocol should give us before executing the transaction. As a result, we’ll uncover an “Easter egg”—a butterfly egg, to be precise—in the fee calculations, which I’ll reveal in Part 4 or 5 of this series.

run()

The run function first gets the number of depletion swaps we’ll need for wETH and osETH in lines 36 and 37, then passes them to the BalancerHacker contract in line 41.

The balances cannot be reduced to MAX_BALANCE_FOR_EXPLOIT in a single swap due to potential overflow errors (as seen in Part 1), so we reduce them over multiple swaps.

getNoOfDepletionSwaps() returns the number of swaps needed to achieve this.

getNoOfDepletionSwaps()

In each loop/swap, we reduce the token balance by 99% until we have at most MAX_BALANCE_FOR_EXPLOIT left in the pool. The hacker also used 99%, which is the most optimal amount we can reduce per swap. It is optimal because it gives us the lowest number of swaps we need to deplete the pool to 51,268.

When we shift a value two places to the right—or divide by 100, as in line 99 above—we are effectively taking 1% of that value.

BalancerHacker Contract

In the deployer, the number of depletion swaps and the value 30 are passed to the constructor of the BalancerHacker contract in line 41.

Thirty represents the number of times we’ll exploit the bug in Step 2. From Part 1, we know that we can increase our profit by exploiting the rounding bug multiple times. We use 30 here because it’s what the original hacker used. As we’ve established in Part 1, it should be at least 2.

We execute the exploit in the constructor, allowing us to deploy and run the attack in a single transaction.

batchSwap() call

Our call to batchSwap() in line 230 above requires an array of all the swaps to be executed in steps 1-3 of our attack plan.

We need to create all the swaps before calling Balancer. As each swap updates the pool state, we must simulate these updates. Keeping an updated state will allow us to generate correct swap values.

To handle this, we implement close replicas of the swap functions we interact with.

_exploit()

Before calling _createDepletionSwaps() in line 56, _exploit() fetches the balances and clones them in lines 38 and 39, respectively.

We will modify the balances during simulation, so we keep a copy of the originals to calculate our profit.

We declare allSwaps in line 42, fetch scaling factors in line 44, and fetch the actual BPT supply in line 51, storing it in bptActualSupply.

BPT has three types of supply:

  • virtual/circulating supply = total amount of BPT held by liquidity providers

  • actual supply = virtual supply + unminted protocol fees

  • total supply = virtual supply + pool balance

The pool balance is the amount of BPT held in the pool to facilitate BPT swaps. It has no value since it does not represent ownership in the pool.

_createDepletionSwaps() returns the depletion swaps and the updated BPT actual supply.

_createDepletionSwaps()

The function runs a loop in line 285 above that returns each swap and its post-BPT actual supply in line 287.

Then it does a sanity check in lines 297 and 302 to ensure the balances are where we want them.

Let’s see how _buildDepletionSwapStep() builds a swap.

_buildDepletionSwapStep()

A depletion swap exchanges BPT for an underlying token; the _swapWithBpt() function in the pool completes this exchange. _buildDepletionSwapStep() simulates the action of this function for each swap.

_swapWithBpt() handles non-proportional joins and follows these three steps in execution:

  1. Collects protocol fees from swaps done since the last join or exit.

  2. Calculates the BPT required for the swap.

  3. Collects protocol fees for the current swap.

Let’s see how _buildDepletionSwapStep() achieves each of the above.

1. Collect Protocol Fees

The protocol fees collected at the beginning of _swapWithBpt() are accumulated protocol fees that have not yet been minted. The table in the Fees section above explains this in the “Issued in Swaps” row. For the first depletion swap, we may have unminted fees. But these unminted fees have already been added to the bptActualSupply passed to _buildDepletionSwapStep(). Remember how we assigned bptActualSupply in the _exploit function?

bptActualSupply = POOL.getActualSupply();

We further explained what it was made up of:

actual supply = virtual supply + unminted protocol fees

We can see that bptActualSupply already includes these fees in the first depletion swap. For subsequent swaps, there are no protocol fees because protocol fees are always issued in non-proportional swaps/joins, as mentioned in the Fees table. This is what happens in step 3 of _swapWithBpt().

2. BPT calculation

The BPT required for the swap, bptAmountIn, is calculated in line 366 by calling _calcBptInGivenExactTokensOut() in the StableMath library. StableMath implements the StableSwap math used to calculate the invariant and the amounts required to complete swaps.

Calculations in StableMath do not treat BPT as a token, but our balances array (which represents the current pool token balances) includes BPT. To address this, we scale the wETH and osETH balances and store them in the scaledBalancesWithoutBpt array in lines 247 and 251, respectively.

amountOut is also scaled in line 339 and stored in scaledAmountsOutWithoutBpt in line 357. This array represents the token amounts we want to swap out for BPT. We initialise only one index in line 357, since each swap outputs either wETH or osETH. The other index remains 0.

The function also uses:

  • preJoinExitInvariant: The invariant of the pool before the swap.

  • SWAP_FEE_PERCENTAGE: Applies liquidity provider fees to the BPT amount.

  • AMPLIFICATION_PARAMETER: used in the curve math to calculate the BPT amount.

After bptAmountIn is calculated, we still need to compute the protocol fee.

3. Protocol Fee Calculation

In line 389, _applySwapAndFees() calculates the protocol fees, adds it to bptActualSupply and returns the new balance.

nextBptActualSupply = bptActualSupply + bptAmountIn + protocolFee

It takes 4 arguments:

  • scaledBalances: The array of current scaled token balances.

  • scaledBalanceChanges: Array of amounts leaving the pool.

  • preJoinExitInvariant: The invariant of the pool before this current swap.

  • bptActualSupply: The actual supply of BPT before this current swap.

Let’s take a look at how _applySwapAndFees() calculates the protocol fee.

_applySwapAndFees() does 2 things: applies balance changes to balances and calculates fees.

In line 819 above, it applies the balance changes to the current balances.

In line 828, it calculates the protocol fee by calling _getProtocolFee().

As mentioned earlier, the swap fee is already included in the BPT amount returned by _calcBptInGivenExactTokensOut(). A portion of that fee is allocated to the protocol.

_getProtocolFee() computes this portion and returns it. The protocol fee is then added to postJoinExitSupply and returned along with the updated balances.

This process gives us the updated state and the information needed to build a swap. We’ll complete the loop in _createDepletionSwaps() to create all the swaps before doing a sanity check.

Final Check

After creating all the swaps, we verify the updated balances in _createDepletionSwaps() to ensure they match MAX_BALANCE_FOR_PROFIT. If they do, _createDepletionSwaps() returns the new BPT actual supply and the depletion swaps.

At this point, we have successfully simulated swaps in the Balancer ComposableStablePool. We performed flash swaps that allowed us to borrow BPT, swap it for pool liquidity, and apply fees.

If you’re wondering why we didn’t explicitly borrow BPT in the code, it’s because the protocol effectively allows us to use BPT we don’t own for the swap. However, since we still have to repay it, this is equivalent to borrowing.
