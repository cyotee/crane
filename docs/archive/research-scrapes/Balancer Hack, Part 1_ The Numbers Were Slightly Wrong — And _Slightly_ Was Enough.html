Balancer Hack, Part 1: The Numbers Were Slightly Wrong — And "Slightly" Was Enough

From Rounding Quirk to Exploit Plan

nonseodion
Apr 06, 2026

November 3rd. The alert hits: Balancer V2 has been hacked. That doesn’t make sense — Balancer has been live for almost five years, and time is supposed to harden code (the Lindy effect). BlockSec tweets, analysts swarm, and the industry tries to force the story into a familiar shape. But the bug isn’t unique to a complex logic flaw or a reentrancy attack. It’s a single stubborn digit at the edge of precision — small enough to ignore, big enough to empty a pool.

Trail of Bits had audited the Balancer codebase and flagged a similar class of rounding issue in a neighbouring contract, but left the severity as Undetermined.

Undetermined is where bugs go to hibernate.

Finding a bug is the easy part. The hard part is making it cashable: turning “a weird rounding edge case” into “I can drain value, on purpose, repeatedly, without the system stopping me.” Until you can show that path, the protocol has every excuse to ignore it. No exploit, no urgency. No urgency, no fix. No fix, no pay.

The Balancer attacker didn’t just spot the bug. They built a bridge from that bug to money — transaction by transaction. They figured out what state the pool needed to be in, how to push it there, and how to harvest the bias.

That’s the bridge I’m going to show you.

This is Part 1 of a five-part series. We’re starting at the moment the bug stops being a curiosity and becomes a plan: how an attacker could identify the rounding quirk, convince themselves it’s extractable, and sketch the strategy that the rest of the exploit will later execute.

Balancer is an AMM, just like Uniswap. It offers different types of pools for trading tokens and uses a single contract called the Vault to communicate with any of its pools, enabling multiple swaps across different pools in one transaction. Liquidity providers deposit tokens to facilitate those swaps — and it was those deposited tokens that were stolen.

I didn’t hack Balancer. But to learn from it, we’re going to think like the only person who mattered that day: the one who turned Undetermined into Critical.

Finding the Bug

As security researchers, we carry a mental catalogue of patterns that alert us to possible bugs in code. One such pattern is pair functions — functions that complement each other, where the action of one is meant to counteract the effect of the other. Deposit and withdraw are a classic example: if you start with a zero balance, deposit $4, then withdraw it, you should return to zero. Other examples include mint/burn, stake/unstake, and lock/unlock.

With that pattern in mind, let’s look at a snippet from ScalingHelpers.sol in the Balancer V2 repo.

There’s a _downscaleUp() and a _downscaleDown() function — but there’s only one _upscale() function. Where are _upscaleUp() and _upscaleDown()?

On closer inspection, _upscale() computes amount * scalingFactor and rounds the result down. Because it scales a value up using a scaling factor and rounds the result down, we could more precisely call it _upscaleDown(). That raises an obvious question: how does the protocol then upscale a value and round it up? In other words, where is _upscale()‘s rounding counterpart?

There’s a comment inside _upscale() that appears to answer that:

Upscale rounding wouldn't necessarily always go in the same direction: in a swap for example the balance of token in should be rounded up, and that of token out rounded down. This is the only place where we round in the same direction for all amounts, as the impact of this rounding is expected to be minimal.

In other words, the developers believed that always rounding down would have a negligible effect. We shall see.

This means there are places in the code that should use _upscaleUp(), but instead use _upscaleDown(). Let’s go looking for them.

Narrowing the Target

_upscale() is used across multiple pool types in Balancer.

Boosted pools, which use Linear pools as a building block, had already been exploited in 2023 and deprecated by Balancer — not enough liquidity there for our purposes.

Weighted pools use _upscale() in the onSwap hook.

In line 42, it upscales balanceTokenIn using scalingFactorTokenIn. However, there’s no rounding error here because the result of balanceTokenIn * scalingFactorIn isn’t floored. We can interpret the _upscale() function as follows:

_upscale(x)=floor(x∗scalingFactor1e18)

For a rounding error to occur, the value to be floored must produce a decimal. This means the fraction portion must return a decimal. If scalingFactor is a multiple of 1e18, the fraction does not result in a decimal. _computeScalingFactor() initialises scalingFactorTokenIn during contract setup. We’ll know if scalingFactor is a multiple of 1e18 from there.

In line 625, the value returned from _computeScalingFactor() is a multiple of FixedPoint.ONE, where FixedPoint.ONE = 1e18. Since the scaling factor is always a clean multiple of 1e18, the division never produces a remainder; hence, there is no rounding error in Weighted pools. On to the next pool.

Composable Stable Pools use _upscale() in three functions: _swapWithBpt, _swapGivenIn, and _swapGivenOut.

Let's start with the low-hanging fruits: _swapGivenIn() and _swapGivenOut(). These two functions handle swapping the tokens deposited as liquidity — the basic purpose of any AMM. _swapWithBpt() handles swapping with BPT (Balancer Pool Token), the token users receive when they deposit into the pool, which represents pool ownership. The pool is called composable because you can swap BPT with any token in the pool, and stable because it supports correlated assets — tokens with similar prices, such as USDC/USDT.

Let’s start with _swapGivenIn.

_upscale() upscales swapRequest.amount in line 60 above. From our Weighted pools analysis, we know the scaling factor must not be a multiple of 1e18 for a rounding error to be possible. Let's verify this in _swapGivenIn(). The scalingFactors array is populated by the _scalingFactors function.

In line 276, the _scalingFactors function shows that each scaling factor is the product of _getScalingFactor() and _getTokenRate(). _getScalingFactor() returns an immutable variable computed by _computeScalingFactor(), which we already know won't help us. But _getTokenRate() fetches its value from a state variable called _tokenRateCaches. It is populated by an external contract called a rate provider that implements the IRateProvider interface. We can't determine the exact value of _tokenRateCaches from the code alone, so let's fetch the rate from a live contract.

Running the script above against the osETH/wETH Composable Stable Pool, we get back a rate of 1063442518645396512 for osETH.

That’s not a multiple of 1e18. Now we’re getting somewhere.

The Right Function: _swapGivenOut

Let's see what this rate means in practice by computing _upscale() with the rate we fetched and an arbitrary swapRequest.amount.

The result is rounded down, losing its fractional portion — exactly what we were looking for.

Now let’s see how this affects each swap function.

In _swapGivenIn, the contract knows how much of a token the user wants to swap in (swapRequest.amount), and uses it to calculate how much they'll receive (amountOut). When _upscale() rounds down swapRequest.amount, the protocol thinks the user is swapping in slightly less than they actually are, so amountOut on line 62 is smaller. We receive fewer tokens. This doesn't help us — we want to extract value from the pool, not lose it. Next function, please.

In _swapGivenOut, the contract knows how much of a token the user wants to receive (swapRequest.amount), and uses it to calculate how much they must deposit (amountIn). When we upscale swapRequest.amount on line 76, the amount used to calculate amountIn in line 81 is smaller than the amount we will actually receive. We end up paying less than we should for the tokens we walk away with.

This is the behaviour we’re looking for. But how do we exploit it?

Planning the Exploit

For this plan, we’ll use the wETH/osETH pool mentioned above. It’s on Ethereum mainnet and was exploited in block 23,717,396.

The naive approach would be to call _swapGivenOut and pocket the difference between what we paid and what we should have paid. The problem is that the difference amounts to a few wei.

I ran that experiment so you don’t have to — 1 WEI profit is too small. We need to fry bigger fish.

We know from rounding exploits like Hundred Finance that the attackers found ways to inflate the value lost to rounding. If I have 1.5 wei and round down to 1 wei, I’ve lost 0.5 wei — completely insignificant. But if that 1.5 wei represented 1.5 BTC, losing 0.5 wei means losing 0.5 BTC.

The key insight: if we can reduce the osETH balance in the pool to a very small number, any rounding loss on that tiny balance will represent a proportionally enormous amount of wETH.

As a rough illustration: if the pool holds 10 osETH and 1,000 wETH, and we try to swap 2.5 osETH for 250 wETH, rounding might mean we only pay ~2 osETH for 250 wETH (The pool uses a Stable Swap curve similar to Curve Finance’s, so the actual numbers require the full curve equation; this rough illustration is just for intuition.).

The real balances before the exploit are 4,922,356,564,867,078,856,521 wETH and 6,851,581,236,039,298,760,900 osETH. After applying their respective scaling factors, these become 4,922,356,564,867,078,856,521 and 7,249,880,088,150,284,833,541 (wETH is unchanged because its rate is 1e18). At these balances, 0.1 osETH is worth only about 0.0679 wETH.

49223565648670788565217249880088150284833541∗0.1=0.06789569627

For 0.1 osETH to be worth 1 wETH, we'd need to reduce the pool to roughly 492 osETH.

49223565648670788565211e18∗0.1=492.2356564867

So our first step would be to remove ~6,851,581,236,039,298,760,434 osETH from the pool.

We used 1.058132398695929516 as the scaling factor and not 1058132398695929516 because we already divided it by 1e18.

Enter batchSwap

Balancer's Vault has a batchSwap function that allows users to execute multiple swaps across different pools in a single transaction, settling all balances only at the end. This is called a flash swap. You can, for example, swap ETH for WBTC on one pool, then swap WBTC for DAI on another — with the Vault collecting the ETH owed only after all swaps are complete. You can even swap the DAI back to ETH to repay what you borrowed.

This mechanism allows us to sequence all our swaps atomically:

  1. Swap wETH for osETH to reduce the osETH balance to ~492.

  2. Swap wETH for 10 osETH to exploit the rounding — 10 * 1.058132398695929516 = 10.581323986959295160, which truncates to 10, meaning the protocol doesn't account for the 0.58 osETH, and charges us ~5e18 wETH less than it should.

  3. Swap osETH back to wETH to repay what we owe from steps 1 and 2, keeping ~5 wETH in profit.

But when we try this, the Vault reverts with BAL#003 — a multiplication overflow error — during the second swap’s invariant calculation. Why? After step 1, the wETH balance required for that state is approximately 1.5 × 10³¹ wETH — an astronomical number, orders of magnitude larger than the total ETH supply (~120.7 million ETH at the time of writing). The Stable Swap curve is designed to price correlated assets at near parity; when balances diverge this wildly, the invariant calculation overflows.

This exploit path is a dead end. Back to the code.

A New Path: Corrupting the Invariant

Let's look more carefully at what happens inside _swapGivenOut. After _upscaleArray() and _upscale() process the balances and swapRequest.amount respectively, they're passed to _onSwapGivenOut() → _onRegularSwap() → StableMath._calcInGivenOut(). (→ represents “calls”).

Inside _calcInGivenOut(), the output token's balance is temporarily reduced in line 174 to calculate the new input token balance, finalBalanceIn, in line 176. The pool assumes the invariant is constant with these new balances — so if the balances end up slightly different from what is assumed, the invariant shifts. In line 185, balances[tokenIndexOut] is updated back to its old balance, so its reduction is not permanent. The permanent balance change is applied later, in line 439 of _processGeneralPoolSwapRequest().

This raises an important question: are the subtractions in _calcInGivenOut() and _processGeneralPoolSwapRequest() equivalent?

First of all, the subtraction done in _calcInGivenOut() and the subtraction done in _processGeneralPoolSwapRequest() are done with upscaled and unscaled balances, respectively. But that’s not a problem since the new unscaled balance is upscaled (remember that it is scaled in _swapGivenOut) before being used to calculate the invariant for the next swap in line 296 below of _onRegularSwap() or is it?

Let’s answer the question.

_processGeneralPoolSwapRequest() subtraction=?_calcInGivenOut() subtraction?

This translates to the equation below.

_upscale(x)−_upscale(y) =?_upscale(x−y)

Where x is the output token balance, and y is the output amount.

_upscale(x)=floor(x∗scalingFactor1e18)

Let a = scalingFactor/1e18.

floor(x∗a)−floor(y∗a)=?floor((x−y)∗a)

If a does not produce a decimal, floor has no effect. So we can ignore it.

(x∗a)−(y∗a)=?(x−y)∗a

After factorising,

(x−y)∗a=(x−y)∗a

But if a produces a decimal, floor has an effect. In our case, scalingFactor is not fully divisible by 1e18; thus a results in a decimal, and floor has an effect. Let’s use 1058132398695929516 as our scaling factor and do two examples that satisfy both conditions.

a=10581323986959295161e18=1.058132398695929516

Let x = 100 and y = 39.

Let x = 18 and y = 17.

If the osETH balance was 18 and we requested 17 osETH as output, the code would assume that the balance should be 2 in _calcInGivenOut(), but it would end up being 1 in _processGeneralPoolSwapRequest(). This would reduce the invariant at the end of the transaction.

At large balances, this discrepancy is negligible. But testing across different powers of 10 reveals a pattern: when one token’s balance is between 10,000 and 100,000, the invariant change becomes both significant and non-reverting. Below 10,000 the invariant change is small; above 100,000 it may revert due to overflow. The sweet spot is the window between those bounds. So we have to make sure our balances stay in this range to cause a good shift in the invariant. We’ve successfully varied the invariant, let’s see how we can use it for our “good”.

The Invariant–BPT Connection

What good does a corrupted invariant do us?

It turns out the invariant is directly tied to BPT (Balancer Pool Token) pricing. In line 1089 above, the getRate() function returns the current BPT price as currentInvariant/actualTotalSupply, where currentInvariant is the current value of the invariant and actualTotalSupply is the supply of BPT tokens representing all tokens in the pool. If the invariant falls without a corresponding decrease in BPT supply, BPT tokens become worth less — meaning the same number of BPT tokens now represents fewer underlying tokens. Equivalently, the same number of underlying tokens can now be exchanged for more BPT.

If we hold BPT (Balancer Pool Token) when we reduce the invariant, we’d end up at a loss. If we hold tokens when we reduce the invariant, we’d receive a higher amount of BPT after invariant reduction. But there’s no upside to that since we will receive the same amount of tokens when we try to exchange the BPT received. So how do we exploit this?

Remember batchSwap()? It allows us to do multiple swaps and pay all the tokens we owe at the end of the swaps. Here’s what we can do with batchSwap:

  1. Use batchSwap() to borrow BPT. Swap BPT tokens for the underlying pool tokens, driving the pool to low liquidity. We now have tokens and owe the Vault BPT.

  2. Exploit the rounding at low liquidity. Perform a _swapGivenOut that triggers the rounding error in _upscale(), corrupting the invariant downward.

  3. Repay the BPT debt — at the new, lower rate. Because the invariant has fallen, we need fewer underlying tokens to pay back the BPT we owe.

  4. Pocket the difference. We exit the batch having spent fewer tokens than we received.

On a simpler note, we are borrowing BPT(Balancer Pool Token) and using it to buy tokens in the pool before we reduce the invariant. After we reduce the invariant, we’ll pay back the BPT tokens we owe. But because the invariant has reduced, we’ll use fewer tokens than we bought from the pool to pay the BPT back and end up with extra tokens.

Let’s test this theory.

After following the steps outlined above, we finally ended up with the profits in the screenshot above. I had to make two swaps in step 2; one swap did not reduce the invariant enough to make a profit. With this change, we know that we can increase the swaps in step 2 further to increase our profits.

Conclusion

We started from a cold read of the code and arrived at a working exploit path. We found a rounding asymmetry, identified which pool type was vulnerable, worked through two dead ends, discovered the invariant–BPT relationship, and assembled a strategy that turns a few wei of truncation into a protocol-draining machine.

It’s been a journey, but we’re only 20% done.

The attacker still had to write the code, protect the transaction from MEV bots, optimise for maximum yield, and safely withdraw the funds. The next three parts of this series will cover exactly that, and the last piece will cover lessons and mitigations. If you want to get ahead, the exploit code is already available on Github.
