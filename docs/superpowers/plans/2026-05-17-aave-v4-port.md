# Aave V4 Port Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Vendor the entire Aave V4 codebase (https://github.com/aave/aave-v4) and all transitive dependencies into the Crane repo, with upstream tests intact, so Crane can compile and execute Aave V4 in-tree. No deduplication against existing Crane vendor copies — that is a separate follow-up.

**Architecture:** Mirror the established Aave v3.6 port (`contracts/protocols/lending/aave/v3.6/`). Vendored source lands at `contracts/protocols/lending/aave/v4/` preserving Aave's upstream `src/` subtree verbatim. Package-private third-party deps from Aave's `src/dependencies/` land at `contracts/protocols/lending/aave/v4/dependencies/` (matches v3.6). Foundry submodules from Aave's `lib/` that are not already in Crane get vendored at `contracts/external/`. Tests land at `test/foundry/spec/protocols/lending/aave/v4/`. Imports get re-pathed so internal references use relative paths and cross-package references use the `@crane/` remapping.

**Tech Stack:** Solidity 0.8.28 (upstream) under Crane's repo-wide 0.8.30 toolchain, Foundry, `git clone` + `cp -R` for vendoring, `find | xargs sed` for import re-paths.

**Pinned upstream:** `https://github.com/aave/aave-v4` `main` branch as of plan date. The clone step records the exact commit SHA in `docs/protocols/lending/aave/v4/VENDOR_PROVENANCE.md` so future dedup work has an unambiguous reference.

**Out of scope (deferred to dedup pass):**
- Replacing `aave/v4/dependencies/openzeppelin*` with Crane's existing `contracts/external/openzeppelin-contracts/`
- Replacing `aave/v4/dependencies/solady` with Crane's `contracts/solady/`
- Replacing `aave/v4/dependencies/weth` with Crane's `contracts/protocols/tokens/wrappers/weth/v9/`
- Collapsing `external/erc4626-tests` if it duplicates other test helpers
- Resolving Aave V4's viaIR requirement against Crane's AGENTS.md "no viaIR" rule (see Task 14)
- Building Crane-native Facet/Repo/Service wrappers around Aave V4 — porting only, no Crane-pattern wrappers in this plan

**Not vendored at all (intentional):** Aave V4 upstream ships several non-Solidity artifact dirs that are not part of the Solidity build surface and do not exist in the existing v3.6 port. This plan skips them. Re-add later if the user explicitly asks.
- `audits/` — third-party audit reports (PDFs)
- `docs/` — protocol documentation
- `resources/` — diagrams and static assets
- `reports/` — generated coverage/gas reports
- `snapshots/` — gas snapshot JSON baselines (would only be useful if we also wire up `forge snapshot` against the port)
- `scripts/` — TypeScript deployment scripts (Crane uses its own deployment harness)
- `.github/`, `.husky/`, `package.json`, `yarn.lock`, `.solhint.json`, `.prettierrc`, `.editorconfig`, `CLA_LICENSE` — repo tooling/CI/legal that does not belong inside Crane

## Conventions used by every copy task

**Source of truth for the upstream tree:** A scratch clone at `/tmp/aave-v4-upstream/` (created in Task 1 and re-used by every subsequent task; deleted in Task 18).

**Copy method:** `cp -R <upstream>/<subdir>/. <crane>/<subdir>/` (trailing `/.` copies contents into an existing directory; preserves file modes; matches what `git mv` would have done if the upstream were already inside the repo).

**Import re-path rules (apply after every copy step):**

| Upstream import form | Rewritten to | Why |
|---|---|---|
| `from 'src/<path>'` or `from "src/<path>"` | `from '@crane/contracts/protocols/lending/aave/v4/<path>'` | Bare `src/` becomes a `@crane/`-rooted absolute path |
| `from 'lib/erc4626-tests/<path>'` | `from '@crane/contracts/external/erc4626-tests/<path>'` | Submodule deps re-rooted under `external/` |
| `from 'forge-std/<path>'` | unchanged | Crane already remaps `forge-std/=lib/forge-std/src/` |
| `from 'erc4626-tests/<path>'` | unchanged | Added as a Crane remapping in Task 3 |
| Relative imports (`../foo.sol`, `./bar.sol`) | unchanged | Relative imports inside the package stay relative — matches the v3.6 port style |

These are the only re-paths needed because Aave's upstream `remappings.txt` only declares `forge-std/` and `erc4626-tests/`, and its bare imports use the `src/...` form.

**Commit cadence:** One commit per task (after compile/tests gate where applicable). Commit message: `vendor(aave-v4): <task summary>`.

## Task 14: Loosen Aave V4 solc pragmas to compile under Crane's default profile

**Files:**
- Modify: every Aave V4 `.sol` file under `contracts/protocols/lending/aave/v4/` that uses the exact pragma `pragma solidity 0.8.28;` — change to `pragma solidity ^0.8.28;` so Crane's `solc = "0.8.34"` satisfies it.
- Modify: `docs/protocols/lending/aave/v4/VENDOR_PROVENANCE.md`

**No new foundry profile.** No `[profile.aave-v4]`. `foundry.toml` is NOT touched. The Aave V4 vendored tree must build under Crane's existing main profile (`solc = "0.8.34"`, no viaIR). The only change in this task is loosening pinned-exact pragmas to their caret form so the main solc satisfies them. Stack-too-deep failures that surface afterwards are handled in Task 14b via structs + wrapper functions.

### Why pragma loosening rather than a scoped profile

- A scoped profile splits the repo into two compilation universes — easy to forget, easy to drift, and breaks the "`forge build`/`forge test` just work" invariant.
- Caret loosening is a minimal, localized change (one line per affected file) that's easy to audit and easy to re-apply if a future dedup pass re-imports upstream.
- It preserves Crane's `solc = "0.8.34"` invariant for the entire repo.

### Step 1: Inventory files needing pragma loosening

Single Bash call: `grep -rlE "^pragma solidity 0\.8\.28;" contracts/protocols/lending/aave/v4 --include="*.sol"`

Capture the list. Task 6-13 implementer's report identified ~23 such files; verify by running the grep.

### Step 2: Loosen each pinned pragma to caret form

Single shell pipeline (one Bash call):
```
find contracts/protocols/lending/aave/v4 -name "*.sol" -print0 | xargs -0 sed -i '' 's|^pragma solidity 0\.8\.28;|pragma solidity ^0.8.28;|'
```

Notes on what this targets:
- Anchored `^pragma` so we only catch the actual pragma line at start-of-line, not anything mid-file.
- Escaped `.` in `0\.8\.28` so each dot isn't treated as a regex wildcard.
- Only catches the exact pinned form `pragma solidity 0.8.28;` — files already on `^0.8.28`, `^0.8.0`, etc. are untouched.

### Step 3: Verify no exact-pin pragmas remain in the v4 tree

Single Bash call: `grep -rE "^pragma solidity 0\.8\.28;" contracts/protocols/lending/aave/v4 --include="*.sol"`

Expected: empty output.

If any other Aave V4 files use a different exact-pin pragma (e.g., `pragma solidity 0.8.27;`), the implementer should report them so we decide whether to loosen those too. Run a broader check (single Bash call):

`grep -rE "^pragma solidity 0\.[0-9]+\.[0-9]+;" contracts/protocols/lending/aave/v4 --include="*.sol" | grep -v "\^"`

Expected: empty (no other exact-pinned pragmas). If non-empty, report the list as a concern but proceed.

### Step 4: Build under Crane's default profile to surface remaining errors

Expected outcome: solc 0.8.28 errors are gone. Remaining errors:

## Task 14b: Refactor stack-too-deep occurrences with structs + wrapper functions

**Files:**
- Modify: each Aave V4 contract listed in the stack-too-deep worklist from Task 14
- Modify (final step): `docs/protocols/lending/aave/v4/VENDOR_PROVENANCE.md` (record completed refactors)

This task resolves every stack-too-deep failure surfaced in Task 14 using the technique AGENTS.md prescribes: group locals into structs, split functions into wrapper + inner helper, and use block scoping to limit lifetimes. **Do NOT enable viaIR.** Each refactor is a small, conservative change to vendored code — drift from upstream is acceptable here because the alternative (viaIR) is forbidden.

### Refactor pattern (apply per failing site)

For each `(file, function)` entry in Task 14's worklist:

1. **Read the failing function.** Count locals + parameters; identify clusters that move together (inputs vs intermediate state).
2. **Introduce a struct OR split.** Pick the lighter option per AGENTS.md:
   - **Struct for parameter clusters:** when 5+ params logically group (e.g., deposit inputs), bundle them in a struct passed by `memory` or `calldata`.
   - **Struct for intermediate state:** when computation produces 4+ locals used together, group into a `State` / `Context` struct.
   - **Wrapper + inner helper:** for functions where the outer signature must remain (e.g., external interface contract), keep the signature; wrap a call to an `_internalImpl(...)` that takes a struct.
   - **Block scoping `{ ... }`:** when a struct is overkill — wrap unrelated local computations in their own blocks so the compiler can spill earlier.
3. **Preserve external behavior.** Public/external signatures, event emissions, state mutations, and revert conditions stay byte-equivalent. Only the function body's organization changes.
4. **Re-compile.** Single Bash call: `forge build --skip test 2>&1 | tail -30`. If the target error is gone and no NEW errors appear, proceed. If a new error appears, the refactor introduced a bug — back it out and try a different decomposition.
5. **Commit.** One commit per file refactored (or per small cluster of co-located refactors, e.g., all sites in one library file). Message: `refactor(aave-v4): resolve stack-too-deep in <file> via <struct|split|scoping>`.

### Iteration loop

- [ ] **Step 1: Pick the next worklist entry**
- [ ] **Step 2: Apply refactor per pattern above**
- [ ] **Step 3: `forge build --skip test` — verify the targeted error is gone**
- [ ] **Step 4: Commit (with `dangerouslyDisableSandbox: true` on git add / git commit)**
- [ ] **Step 5: Repeat until the worklist is empty**

### Final step

- [ ] **Step 6: Mark the worklist resolved in provenance**

Update `docs/protocols/lending/aave/v4/VENDOR_PROVENANCE.md`:
- Move each entry from `## Stack-too-deep worklist (Task 14b)` into a new `## Stack-too-deep resolutions` section, annotated with the commit SHA and the refactor technique used.
- Confirm a clean `forge build --skip test` and record that fact.

Commit:
1. `git add docs/protocols/lending/aave/v4/VENDOR_PROVENANCE.md` (with `dangerouslyDisableSandbox: true`)
2. `git commit -m "docs(aave-v4): record stack-too-deep refactor resolutions"` (with `dangerouslyDisableSandbox: true`)

### Escalation criteria

If a refactor would require:
- Changing an external/public function signature (which would break downstream callers including Aave's own tests)
- Modifying observable contract state layout
- Adding a new dependency / library not already in the vendored tree

— STOP and escalate to the controller. The decision is whether to drift further from upstream or accept that this contract is left out of the port. Don't make that call inside the implementer.
