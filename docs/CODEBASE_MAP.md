## System Overview

Crane is a **Diamond-first (ERC2535)** Solidity development framework for modular, upgradeable smart contracts. It emphasizes deterministic CREATE3 + callback-based Diamond deployments, a strict Facet-Target-Repo + Service + DFPkg pattern, rich testing (TestBase/Behavior/handler + invariants), and first-class ports of major DeFi protocols with shared dependencies remapped to Crane equivalents.

## Module Guide

### Core Framework & Patterns

**StyleGuide + Init services**
- `contracts/StyleGuide.sol`: Mandatory conventions (headers, import groups, naming `_layoutStruct`/`_onlyXxx`/`param_`, NatSpec + AsciiDoc tags, storage slot hierarchy).
- `InitDevService` / `InitBcService`: Single source of truth for bootstrapping Create3Factory + DiamondPackageCallBackFactory + canonical facets in tests/scripts (BattleChain variant for traceable top-level).

**Access Control**
- `operable/`: OperableRepo (guards + global/per-function operators), Target, Facet, Modifiers. Guards callable from other Repos.
- `ERC8023/`: MultiStepOwnable (initiate/confirm flow) with buffer + view facet. Full FTR + TestBase.
- `reentrancy/`: Transient (EIP-1153) lock.

**Factories & Deployment**
- Two-factory system: Create3Factory (facets, packages, any contract) + DiamondPackageCallBackFactory (proxies with delegatecall `initAccount`).
- `IDiamondFactoryPackage` (DFPkg): `PkgInit` (immutables), `facetCuts`, `diamondConfig`, `calcSalt`, `initAccount`, `postDeploy`.
- FactoryService libs for grouped deterministic deploys (name-hash salts + `vm.label`).
- `MinimalDiamondCallBackProxy` + resolution via ERC2535 loupe.

**Introspection (Diamond)**
- ERC165Repo + Facet.
- ERC2535Repo (AddressSet + per-facet selectors; diamondCut logic with validation).
- DiamondLoupeFacet + DiamondCutFacet.
- ERC8109 extension.
- Full Behavior/TestBase support + comparators.

**Registries**

**Dedicated Registries Documentation (LR-2 GitBook requirement)**: The registries (FacetRegistry, DiamondFactoryPackageRegistry/PackageRegistry, CallTargetRegistry) are foundational to Crane's deterministic, reusable deployment model. They provide on-chain discovery, canonical resolution, and dynamic configuration. This enables agents and developers to "deploy once, attach everywhere" (LR-4). 

To set up a chain presence (per LR-2): Use the CREATE3 Package (Create3FactoryDFPkg) to deploy your own Create3Factory on a new chain. The Diamond Package Factory (DiamondPackageCallBackFactory, interface ID from central values: `0x949da331`) does **not** need to be redeployed per chain — it is safe and intended for public reuse across all deployments (attach via `setDiamondPackageFactory` or reuse in calc/deploy flows). See cross-links below and `docs/deployment/create3.md`, `docs/deployment/dfpkg.md` (required GitBook areas).

The registries provide on-chain discovery, canonical resolution, and configuration for facets, packages, and call targets. They are core to Crane's reuse model (see LR-4 in PRD.md): deploy facets/packages once via CREATE3, register them, then discover and reuse via queries or canonical lookups instead of hardcoding addresses or redeploying bytecode. This reduces gas and eliminates per-project reimplementation risk. Security benefit (verbatim LR-4): "The primary security advantage is the ability to reuse already deployed and verified code. ... Reusing battle-tested, already-audited deployed logic removes that class of error." Cost benefit: "directly saves gas by simply not needing to deploy as much bytecode."

There are three primary registries (all using Facet-Target-Repo or direct + protected by Operable/ownership; integrated with Create3Factory and bootstrap; facets implement `IFacet` using ONLY central NatSpec values: `facetName()` 0x5b6f4d01, `facetInterfaces()` 0x2ea80826, `facetFuncs()` 0x574a4cff, `facetMetadata()` 0xf10d7a75):

- **FacetRegistry** (`contracts/registries/facet/`): Tracks deployed `IFacet` implementations. Full FTR (IFacetRegistry + FacetRegistryTarget + FacetRegistryFacet + FacetRegistryRepo + FacetRegistryService + Behavior/Handler for testing).
  - **Purpose**: Enables lookup of facets by name, supported interface(s), or function selector(s). Provides `canonicalFacet(bytes4 interfaceId)` to resolve the preferred implementation for a given interface (used heavily during DFPkg construction and Init*Service bootstrap to avoid address literals). Supports `registerFacet` and `setCanonicalFacet` for manual/override registration. Enables agent-proof reuse of verified facets.
  - **Population**: Auto-populated on successful `deployFacet*` calls through `Create3Factory` (or the registry's own deploy wrappers) — after CREATE3 deployment, the code calls `facet.facetMetadata()` (which returns name + `facetInterfaces()` + `facetFuncs()`) and stores via `FacetRegistryRepo._registerFacet`. Manual registration also available via `registerFacet(IFacet, string, bytes4[], bytes4[])`. See Create3Factory._registerFacet (uses `facet.facetMetadata()`), FacetRegistryService. Post-deploy, `Behavior_IFacetRegistry` + tests assert population (LR-7).
  - **Usage/Consumer Interaction**:
    - Query: `allFacets()`, `facetsOfName(name)`, `facetsOfInterface(interfaceId)`, `facetsOfFunction(selector)`, `nameOfFacet(facet)`, `interfacesOfFacet(facet)`, `functionsOfFacet(facet)`.
    - Canonical resolution: `canonicalFacet(someInterfaceId)` — e.g. during package init to pull `diamondCutFacet`, `operableFacet`, etc. Example from InitDevService: `IFacetRegistry(address(factory)).canonicalFacet(type(IDiamondCut).interfaceId)`.
    - In practice (from InitDevService): `IFacetRegistry(address(factory)).canonicalFacet(type(IFacetRegistry).interfaceId)` and similar for `type(IOperable).interfaceId`, `type(IDiamondCut).interfaceId`, etc. Also used for registry facets themselves.
    - The registry facet itself implements `IFacet` (central values: `facetName()` selector `0x5b6f4d01`, `facetInterfaces()` `0x2ea80826`, `facetFuncs()` `0x574a4cff`, `facetMetadata()` `0xf10d7a75`).
    - In tests/agents: Use `Behavior_IFacetRegistry` (expect/isValid paths) + assert post-deploy registry entries.
    - Cross-link: `docs/deployment/dfpkg.md`, `docs/deployment/create3.md` (GitBook areas), `AGENTS.md` (FactoryService + bootstrap + key files), `contracts/registries/facet/FacetRegistryRepo.sol` (uses `DEFAULT_SLOT = bytes32(uint256(keccak256(abi.encode("crane.registries.facets"))) - 1)` per LR-6), `contracts/interfaces/IFacetRegistry.sol`.

- **DiamondFactoryPackageRegistry** (aka Package Registry, `contracts/registries/package/`): Tracks `IDiamondFactoryPackage` (DFPkg) deployments. (IFacet + IDiamondFactoryPackageRegistry + ... + DiamondFactoryPackageRegistryRepo + Behavior_IDiamondFactoryPackageRegistry).
  - **Purpose**: Similar discovery for packages by name, implemented interfaces, or contained facets. Provides canonical resolution per interface for packages. Enables consumers to find packaged facet bundles for deterministic proxy deployment via `DiamondPackageCallBackFactory`. Core to reuse of DFPkgs (e.g. ERC20DFPkg, Create3FactoryDFPkg, CallTargetRegistryDFPkg, BountyBoardDFPkg).
  - **Population**: Auto on `deployPackage*` via Create3Factory (calls `package.packageMetadata()` returning name/interfaces/facets, then `DiamondFactoryPackageRegistryRepo._registerPackage`). Also supports `registerPackage` + `setCanonicalPackage`. See Create3Factory._registerPackage + central DFPkg values: `packageName()` `0xabc8b346`, `facetInterfaces()` `0x2ea80826`, `initAccount(bytes)` `0x870d4838`, `postDeploy(address)` `0x70068fcf`, `facetCuts()` etc.
  - **Usage**: `allPackages()`, `packagesByName`, `packagesByInterface`, `packagesByFacet(facet)`, `canonicalPackage(interfaceId)`, `nameOfPackage`. Used in bootstrap flows (see InitDevService deploying Create3DFPkg/CallTargetDFPkg/BountyBoardDFPkg via `deployCanonicalPackageWithArgs` + `type(ICREATE3DFPkg).interfaceId`) and by agents to reuse shared DFPkgs without redeploying facets. Explicit note per LR-2: The `DiamondPackageCallBackFactory` (interface ID from central: `0x949da331`) does **not** need redeployment per chain — it is safe and intended for public reuse across deployments (see `DiamondPackageCallBackFactory` + `IDiamondPackageCallBackFactory`; e.g. `factory.setDiamondPackageFactory(diamondFactory)` once per Create3Factory instance).
  - Cross-link: `IDiamondFactoryPackage` (central values listed above + `facetCuts()` `0xa4b3ad35`, `diamondConfig()` `0x65d375b3`, `calcSalt(bytes)` `0xd82be56e`), `docs/deployment/dfpkg.md`, `docs/deployment/create3.md`, `AGENTS.md`, `Behavior_IDiamondFactoryPackageRegistry`.

- **CallTargetRegistry** (`contracts/registries/target/`, split into Management + Query facets + DFPkg `CallTargetRegistryDFPkg`):
  - **Purpose**: Acts as a dynamic configuration oracle for call targets. Allows setting default (or per-caller) target addresses for a given interface ID. Consumers query to resolve "who should I call for this interface" without embedding addresses or redeploying when targets change. Useful for proxies, routers, or multi-target systems. Complements other registries for full DI in protocols.
  - **Population**: Explicitly via `setDefaultCallTargetForID` / `setCallTargetForIDForCaller` (governed by management facet, typically owner/operator). Not auto-populated on deploy. Part of bootstrap via `CallTargetRegistryDFPkg` (see InitDevService using canonical facets).
  - **Usage/Interaction**: `defaultCallTargetForID(interfaceId)`, `callTargetForIDForCaller(interfaceId, caller)`. Interfaces split: `ICallTargetRegistryQuery` + `ICallTargetRegistryManagement`. Deployed/queried in core flows and tests (Behavior_ICallTargetRegistry*).
  - Note: Current Repo uses direct keccak slot (see LR-6 migration notes in PRD); query/management separated for access control. Uses central IFacet selectors on its facets.

All registries are deployed as facets (via `Create3FactoryBootstrap` using salts from type names, e.g. `FACET_REGISTRY_FACET_SALT = keccak256(abi.encode(type(FacetRegistryFacet).name))`) and attached to the canonical Create3Factory instance (which itself exposes registry facets). After any factory/package-based deployment, tests must verify that FacetRegistry, PackageRegistry, CallTargetRegistry contain the expected entries (per LR-7). See `contracts/interfaces/IFacetRegistry.sol`, `IDiamondFactoryPackageRegistry.sol`, `ICallTargetRegistry*`, `InitDevService.sol`, `Create3Factory.sol`, and `Behavior_*` (e.g. for registries) for details. Agent usage: Consult `crane-deployment` skill + AGENTS.md for bootstrapping + registry queries in DFPkg flows. Cross-links to GitBook required areas: deployment docs, protocol TestBases (which assert registry state), AGENTS.md.

**General Utilities (and Type Libraries) — Detailed Coverage (LR-2)**

Crane's `contracts/utils/` provides low-level building blocks used throughout Repos, Services, tests, and protocol ports. These are **not** simple wrappers in many cases — they are optimized for Diamond storage patterns, gas, and the set/AMM/permitting needs of DeFi. Detailed coverage supports agent consumption of reusable math/collections (ties to LR-4 reuse: reuse `ConstProdUtils` + Sets instead of reimplementing in every port or protocol integration).

- **Collections & Sets** (`utils/collections/sets/` and `BetterArrays.sol`):
  - Type-specific set implementations (all follow same 1-indexed struct + Repo pattern for storage-direct ops):
    - `AddressSet` / `AddressSetRepo`: struct { mapping(address=>uint256) indexes; address[] values; } (0 = absent). `_add`/`_remove` (idempotent), `_contains`, `_length`, `_index`, `_indexOf`, `_values` (returns storage pointer), `_asArray`, `_range`, `_addAsc`/`_removeAsc` (binary search + shift for sorted determinism), `_sortAsc`.
    - `Bytes32Set` / `Bytes32SetRepo`, `Bytes4Set` / `Bytes4SetRepo` (heavily used for selector sets in ERC2535Repo + comparators), `StringSet` / `StringSetRepo`, `UInt256Set` / `UInt256SetRepo`.
  - **Repo pattern**: Library with dual overloads where applicable; operates directly on `Storage` structs (no copying like OZ EnumerableSet). Supports multi-value add/remove loops. Asc variants preserve order for deterministic enumeration (key for registries, facets).
  - Usage examples: `FacetRegistryRepo` stores `AddressSet` per name/interface/func + allFacets (see `_registerFacet`); `ERC2535Repo` uses `AddressSet facetAddresses` + per-facet `Bytes4Set` for selectors; `OperableRepo` for function operators; `DiamondFactoryPackageRegistryRepo` for packages; handlers/comparators (e.g. `Bytes4SetComparatorRepo`); tests track expected sets.
  - Why custom: Storage pointer returns for gas (e.g. `_values`), direct storage mutation, asc ordering, multi-ops. 
  - Cross-links: Used in almost every Repo (see `ERC2535Repo`, `FacetRegistryRepo`, `OperableRepo`, registry repos), test comparators (`Bytes4SetComparator`), handlers. Documented in AGENTS.md (architecture + testing), `StyleGuide.sol`, protocol services. See `crane-architecture` skill. LR-2 GitBook tie: general type libraries section + test usage via CraneTest handlers.

## Data Flow / Key Usage

**Diamond deployment (typical)**:
InitDevService → deploy facets via Create3Factory (name salt) → build DFPkg with facet refs → diamondFactory.deploy(pkg, pkgArgs) → callback `initAccount` (cuts + pkg init).

**Protocol consumption**:
- Simple: direct interface calls from custom facets.
- Structured: *AwareRepo for address injection + *Service for complex flows (with structs).
- Yield/collateral: RateProvider + ERC4626.
- Gasless: Permit2Aware.

**Testing**:
Inherit TestBase chain → deploy or attach SUT → use Behavior for interface checks or handlers + `invariant_*` for stateful fuzz.

## Conventions (from AGENTS + StyleGuide)

- Naming: `_layoutStruct`, `_initialize`, `_onlyXxx` (in Repo), `onlyXxx` (modifier), `param_`, `layoutStruct`.
- Storage: Hierarchical keccak (`"crane.access.operable"`, `"eip.erc.2535"`, `"protocols.dexes.balancer.v3..."`).
- No viaIR; use structs for complex functions.
- NatSpec + `// tag::Symbol[] ... // end::Symbol[]` on documented symbols.
- `@custom:selector` / `@custom:signature` etc. computed via cast.
- `vm.label` everything deployed.
- Test infra co-located; specs mirror structure.
- BattleChain gate before mainnet for factories/DFPkgs.

## Gotchas

- Vendored code is large — focus edits on `contracts/` non-external and `test/foundry/spec/`.
- Many "ports" are faithful upstream with import remaps only (no new Crane wrappers yet for all).
- Optimizer runs=1 (contract size). Stack-too-deep = refactor with structs.
- Pre-mainnet deploys **must** survive BattleChain attack window.
- Sky/Permit2/ERC4626/Reliquary and Balancer show the "target" level of Crane surface for new ports.
- External/ + lib/ should not be edited directly; track pins/tags.

## Navigation Guide

**Add a new Diamond feature**:
- Define interface + Repo (storage + dual layout + logic) + Target + Facet (+ Modifiers if access).
- Add to a FactoryService for canonical deploy.
- Create DFPkg if packaged.
- Add Behavior/TestBase + spec.

**Integrate a new DEX/lending primitive**:
- Vendor faithful code to `external/<lib>/` (remap shared deps to @crane).
- Add interfaces.
- Write *Service (and *AwareRepo if state/DI needed).
- Add TestBase + fork parity tests.
- Update CODEBASE_MAP.

**Bootstrap in test/script**:
- `InitDevService.initEnv(this)` or BC variant.
- Use returned factories.

**Run tests**:
- `forge test`
- Targeted: `--match-path test/foundry/spec/protocols/dexes/...` or `--match-test testFoo`
